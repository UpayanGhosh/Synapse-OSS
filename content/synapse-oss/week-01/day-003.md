# Day 3 — 8 stages between your message and the AI's response

## Script (90 seconds)

I count 8 stages between your message and my AI's reply.

Not 1. Not 3. 8. And each one exists because something broke without it.

When a message arrives — from WhatsApp, Telegram, Discord, or Slack — the first stop is the FloodGate. A 3-second buffer per user. WhatsApp users send 5 messages in a row instead of one complete thought. Without this, each fragment triggers a separate LLM call. The FloodGate waits, then releases them as one batch.

Stage 2 is the deduplicator. A 5-minute TTL cache of message hashes. WhatsApp's delivery protocol can re-deliver the same message. Without dedup, the AI replies twice to the same question.

Stage 3 is the task queue. An asyncio.Queue with a hard cap of 100 items. When full, new messages get rejected — explicit backpressure instead of silent memory growth.

Stages 4 and 5: two workers running concurrently. Each task runs through Soul-Brain Sync context assembly, hybrid RAG memory retrieval, the dual cognition inner monologue, and the LLM call.

Stage 6 is the LLM. Stage 7 formats the response. Stage 8 sends it back through whichever channel the message came from.

None of this was designed upfront. The FloodGate came after fragmented-reply complaints. The deduplicator after double-response reports. The bounded queue after an OOM crash.

The architecture grew into 8 stages. I didn't plan it that way.
