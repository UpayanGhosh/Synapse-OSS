<objective>
Fix Discord and Slack inbound pipeline by replacing the broken direct task_queue.enqueue registration with flood.incoming() adapter closures in api_gateway.py.

Purpose: Discord and Slack channels currently register with enqueue_fn=None (Discord) or enqueue_fn=task_queue.enqueue (implied by the absent kwarg) — both paths silently drop all inbound messages. Even if enqueue_fn were set to task_queue.enqueue, it would crash because DiscordChannel and SlackChannel both call self._enqueue_fn(channel_msg) passing a ChannelMessage, but task_queue.enqueue() expects a MessageTask and immediately accesses task.task_id at queue.py:45. The fix is the same pattern WhatsApp uses: define per-channel async adapter closures in api_gateway.py that call flood.incoming(chat_id, text, metadata) and inject those as enqueue_fn.

Output: api_gateway.py with correct Discord and Slack adapter closures; extended tests in test_discord_channel.py and test_slack_channel.py confirming the adapter routes to flood.incoming(), not task_queue.enqueue directly.
</objective>

```python
async def _make_flood_enqueue(channel_id: str):
    """
    Factory: returns an async callable that routes a ChannelMessage through
    FloodGate (with dedup) instead of passing it directly to task_queue.enqueue().
    CHAN-05 reference pattern: see unified_webhook() which calls flood.incoming().

        if dedup.is_duplicate(channel_msg.message_id or ""):
            return

            message=channel_msg.text,
            metadata={

                "sender_name": channel_msg.sender_name,

            },
        )
    return _enqueue
```

```python
    shape satisfies the adapter contract.
    """

    def _make_flood_adapter(self, collected):
        """Returns an async enqueue_fn that captures calls to flood.incoming()."""

                "text": channel_msg.text,
                "message_id": channel_msg.message_id,
```
